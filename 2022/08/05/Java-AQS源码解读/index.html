AQS

AbstractQueuedSynchronizer，抽象的队列同步器，是用来构建锁或者其他同步器组件的重量级基础框架及整个JUC体系的基石，通过内置的FIFO队列来完成资源获取线程的排队工作，并通过一个int类型变量表示持有锁的状态。

¶概述

AQS使用一个volatile的int类型的成员变量来表示同步状态，通过内置的FIFO队列来完成资源获取的排队工作，将每条要去抢占资源的线程封装成一个Node节点来实现锁的分配，通过CAS完成对state的修改。

如果共享资源被占用，就需要一定的阻塞等待唤醒机制来保证锁分配。这个机制主要用的是CLH队列的变体实现的，将暂时获取不到锁的线程加入到队列中，这个队列就是AQS的抽象表现。它将请求共享资源的线程封装成队列的节点，通过CAS、LockSupport的方式维护state的状态，使并发达到同步的控制效果。

¶ReentrantLock

从ReentrantLock源码入手，解读AQS

ReentrantLock和Synchronized是典型的可重入锁。因为在执行对象初始化时，如果不填入true参数，默认创建的是非公平锁。从其两个构造函数就可以看出。

sync是什么类型的变量？sync是ReentrantLock的私有变量，其类型是一个抽象的静态内部类，它继承自AQS。

¶AbstractQueuedSynchronizer

Sync与NonfairSync都继承自AbstractQueuedSynchronizer

AQS提供了对阻塞式锁以及同步器最基础的框架，其依靠FIFO等待队列来实现，上文提到的CLH队列，本质是一个由双向链表且携带哑元节点构成的队列，后续会进行分析。

AQS中保存了实现同步的关键信息。

state是表示同步状态的关键变量，具体可以理解为，当前有多少线程获取到了锁；head与tail记录了CLH队列的头尾指针，具体实现都在静态内部类Node中。

¶Node

Node 是CLH队列中的节点，其内部保存的value是正在排队获取锁的线程。prev，next记录了前后节点的信息。waitStatus与aqs的state不同，它代表的是尝试获取锁的线程的状态，其值为负数是，表示不需要被调度，可能被其他线程阻塞需要唤醒，或者该线程已经放弃抢占。

¶AQS分析

以三个线程抢占一把锁为例，对AQS底层进行分析。

假设线程A一定是第一个抢占到锁的，也就是说，第一次执行lock()方法。ReentrantLock的lock()调用的是内部类Sync的lock()，但是Sync的lock()是一个抽象方法，从架构图可以看出，Sync类下有两个实现类FairSync与NonFairSync，ReentrantLock默认创建的是非公平锁，所以先关注NonFairSync的lock()。

这个方法很简单，如果不了解CAS的话可以去了解一下。compareAndSetState是AQS内部实现的方法，通过CAS将state的值从0设为1。0表示没有线程占用锁，1表示有线程占用锁。如果CAS成功改变了state的值，那么就将当前占有资源的线程设置为当前线程。否则进入acquire()分支。

由于我们假设，线程A一定是第一个获得锁，所以CAS逻辑一定为true，暂且不看acquire分支。其实到这里，线程A已经完成了加锁的工作。实际上FairSync与NonFairSync的差别并不大。

¶CLH队列分析

并发环境避免不了资源争夺与抢占。在线程A没有释放资源的前提下，线程B不应该持有资源，并且保持等候状态。下面就来看AQS对于抢占与等候是怎么实现的。

经过上文的分析，线程B也会执行lock()方法，并且CAS不会返回true，因为此时线程A占用资源，state值为1。换句话说，如果在线程B抢占资源之前，线程A已经释放资源，那么线程B也不需要进行抢占，可以直接获取到资源。

那么接下来就是对acquire()分支的分析。

acquire()是AQS提供的方法，其中有三个主要函数，此时传递的arg参数的值为1：

  • tryAcquire(int arg)
  • addWaiter(Node mode)
  • acquireQueued(final Node node, int arg)

这三个方法都是AQS提供的，tryAcquire有点特殊，AQS类没有提供默认实现，而是直接报错。

这意味着，AQS强迫其实现类实现该方法，那么就去其实现类NonFairSync看。

为什么acquire()还要再尝试获取一遍资源？想象一个场景，线程B在第一次尝试获取A占有的资源时获取失败了，此时正好线程A释放了资源。线程B也应当直接获取资源，而不是进行繁琐的等待操作。这里的再一次获取就对应了这个场景。

再次获取失败之后，就要进行入队等待操作了。

node为当前要添加入等待队列的节点，pred为AQS的尾指针。如果pred不为空，指向一个简单的双向链表的尾插入操作，CAS保证并发修改的正确性，以及前后指针的变动。为空的话指向enq(node)逻辑。由于线程B是第一个加入等候队列的，此时AQS的tail没有任何指向。

enq整体是一个自旋的操作，变量t也是指向的AQS的tail尾指针，如果尾指针为空，通过CAS为CLH队列设置了一个空的头节点。

compareAndSetHead插入的就是一个Node()空节点，也就是上文提到的哑元节点。这样做的好处是避免了一些繁琐的边界判断操作，通常为链表设置一个虚拟的头节点是一个非常有用的操作，尽管会浪费一点空间，这点空间可以说是微乎其微，但会是编码简单许多。

最后一步调用的是acquireQueued，整体还是一个自旋操作。p是当前插入节点的前一个节点，如果前一个节点为头节点，那么让当前节点尝试再去获取资源。若成功获取，就将当前节点设置为新的哑元节点，将前一个哑元节点删除。否则就调用LockSupport的park()将线程阻塞。

shouldParkAfterFailedAcquire主要任务是判断该节点状态，是否能够被阻塞。最主要的操作是parkAndCheckInterrupt

此时，线程B已经成功被添加到了CLH队列中，等待当前持有资源的线程释放资源。后续线程C也是如此，不再额外演示。

¶释放资源及调度

线程A在执行完自己的业务代码后，释放资源，供其他线程使用。业务代码中具体为unlock()方法。与lock()相类似，其调用的都是抽象类Sync类的实现类的release()。具体的调用链为：

  1. AQS类的release
  2. Sync类的tryRelease
  3. AQS类的unparkSuccessor

tryRelease主要工作将AQS中的state数量递减，以及可重入锁的实现。如果AQS的state为0，表示资源已经被释放，把当前占有资源的线程设置为null。这里还并没有去唤醒CLH队列中的节点去抢占资源。

h指向的是CLH队列的头节点，unparkSuccessor对应将下一个等待线程唤醒

s指向的是下一个需要唤醒的节点。然后调用LockSupport的unpark()将其唤醒。至此unlock()解锁工作就全部完成了。

那么被唤醒的线程如何获取锁？回到加锁时候的acquireQueued方法，上文提到这是一个自旋操作，被唤醒的线程会继续执行这个自旋操作，尝试获取资源。

¶公平与非公平

经过上述的分析，对加锁与解锁的流程大致介绍了一遍。这里要提出一个疑问：ReentrantLock默认创建的是非公平锁，而底层却是一个队列，非公平锁表述的是获取资源的线程不会按照申请的先后顺序。那么AQS哪里体现了非公平锁？

在分析获取资源时提到过一个场景：线程B在第一次尝试获取A占有的资源时获取失败了，此时正好线程A释放了资源。

类似的场景：假设等待队列中有B、C两个线程，A线程持有资源。在A线程释放资源的那一刻，恰巧线程D来申请资源。还没等到唤醒B线程，D线程已经获取到了资源，那么线程B只能继续等待。很明显，这种资源的获取方式显然不满足公平锁的定义。

那么AQS是如何实现公平锁的呢？FairSync与NonFairSync的区别仅在对于tryAcquire方法的实现，lock方法直接调用了acquire，也就是FairSync自身实现的tryAcquire。

更进一步说，公平与非公平的差异仅仅只有hasQueuedPredecessors的判断。

用简单的话来概括AQS是如何实现公平锁的：只有当前AQS的state为0，并且尝试获取资源的线程是CLH队列的头节点，才能够获取到。

¶总结

AQS是同步器最底层的抽象，上层实现ReentrantLock、CountDownLatch、Semaphore都源于AQS。其内部都由Sync实现，不同点就在于针对不同场景，实现方式不同。
